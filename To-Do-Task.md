# React — завдання (To-Do застосунок)

## Мета
Реалізувати простий To-Do застосунок з використанням **React** та **React Router**, який демонструє коректну роботу з **хуками**, **маршрутизацією**, **CRUD-операціями** та **збереженням стану в localStorage**.

---

## Технічні вимоги
- React (лише функціональні компоненти)
- React Router (`react-router-dom`)
- UI-бібліотеки не обовʼязкові
- Дозволено JavaScript або TypeScript

---

## Маршрути
Застосунок повинен містити такі маршрути:

| Маршрут | Опис |
|------|------|
| `/` | Список задач |
| `/new` | Створення задачі |
| `/task/:id` | Деталі задачі |
| `/task/:id/edit` | Редагування задачі |
| `*` | Сторінка «Не знайдено» |

---

## Модель задачі
Кожна задача повинна містити щонайменше:

- `id` — рядок
- `title` — рядок
- `completed` — boolean
- `createdAt` — number (timestamp)

Опціонально:
- `description` — рядок

> Редагування має підтримувати зміну `title` та `description`.

---

## Стан та збереження даних
- Усі задачі мають зберігатися у React-стані на рівні застосунку (наприклад, у `App` або Context).
- Дані повинні зберігатися між перезавантаженнями сторінки за допомогою **localStorage**.
- Під час старту застосунку задачі зчитуються з localStorage.
- Будь-яка зміна задач повинна оновлювати localStorage.
- Рекомендований ключ: `todo_tasks_v1`.

---

## Вимоги до хуків
Рішення **обовʼязково** повинно використовувати:

- `useState`
- `useEffect`
- `useParams`
- `useNavigate`
- `useLocation`

Опціонально (бонус):
- `useRef`
- `useMemo`
- `useCallback`

---

## Функціональні вимоги

### 1) Список задач (`/`)
- Відображати список усіх задач.
- Показувати:
    - загальну кількість задач
    - кількість виконаних задач
- Кожна задача повинна:
    - відображати назву
    - дозволяти перемикати статус виконання
    - відкривати сторінку деталей при кліку
- Повинна бути навігація на `/new`.

---

### 2) Створення задачі (`/new`)
- Форма повинна містити:
    - обовʼязкове поле `title`
    - опціональне поле `description`
- Валідація:
    - `title` не може бути порожнім після `trim()`
- Після сабміту:
    - створюється нова задача
    - відбувається редірект на:
        - `/` **або**
        - `/task/:id`

---

### 3) Деталі задачі (`/task/:id`)
- Відображати інформацію про задачу:
    - назву
    - статус виконання
    - дату створення
    - опис (якщо є)
- Обробити ситуацію, коли задача не знайдена.
- Забезпечити можливість:
    - змінити статус виконання
    - видалити задачу (з підтвердженням)
- Повинні бути:
    - кнопка переходу до редагування (`/task/:id/edit`)
    - кнопка **«Назад»** (див. вимогу щодо route state)

---

### 4) Редагування задачі (`/task/:id/edit`)
- Форма редагування повинна бути попередньо заповнена поточними даними задачі:
    - `title` (обовʼязкове)
    - `description` (опціональне)
- Валідація:
    - `title` не може бути порожнім після `trim()`
- Дії:
    - **Зберегти** — оновити задачу та перейти на `/task/:id`
    - **Скасувати** — скасувати зміни та повернутися назад
- Необхідно коректно обробити ситуацію, коли задача не знайдена.

---

## Вимога щодо route state
- При переході зі списку задач (`/`) на сторінку деталей (`/task/:id`) необхідно передавати попередній маршрут через **route state**.
    - Приклад: `{ from: location.pathname + location.search }`
- Кнопка **«Назад»** на сторінці деталей повинна:
    - повертати користувача на попередній маршрут, якщо він є
    - інакше повертати на `/`

---

## Опціональні покращення (не обовʼязково)
- Пошук задач за назвою
- Приховування виконаних задач
- Автофокус на інпутах створення/редагування
- Попередження про незбережені зміни при виході з редагування
